Great question — this is the core concept that makes Next.js different. Let me explain the full flow of both.

React (Vite) — Client Side Rendering (CSR)

You type: http://localhost:5173
              ↓
        Browser sends request to server
              ↓
        Server sends back index.html
        (almost empty — just <div id="root">)
              ↓
        Browser downloads JavaScript bundle
        (your entire app code — can be 1-5MB)
              ↓
        JavaScript RUNS in the browser
              ↓
        React builds the page in the browser
              ↓
        User finally sees the page
What the server actually sends:


<!doctype html>
<html>
  <body>
    <div id="root"></div>   ← EMPTY, nothing here yet
    <script src="/main.jsx"></script>  ← browser downloads this
  </body>
</html>
Then JavaScript runs and fills it:


<div id="root">
  <nav>...</nav>
  <h1>Hello World</h1>   ← built by React in browser
  <p>...</p>
</div>
Problems with this:

User sees a white blank screen while JS downloads and runs
Google crawler sees empty <div id="root"> — bad for SEO
Slow devices struggle because all work is done on the browser
Next.js — Server Side Rendering (SSR)

You type: http://localhost:3000
              ↓
        Browser sends request to Next.js server
              ↓
        Next.js server runs your layout.jsx + page.jsx
        (your React code runs on the SERVER, not browser)
              ↓
        Server builds the COMPLETE HTML page
              ↓
        Server sends FULL HTML to browser
              ↓
        User sees the page IMMEDIATELY
              ↓
        JavaScript downloads in background
              ↓
        React "hydrates" — attaches event listeners
        (buttons, clicks, useState now work)
              ↓
        Page is now fully interactive
What the server actually sends:


<!doctype html>
<html>
  <body>
    <nav>My Navbar</nav>
    <h1>Hello World</h1>    ← ALREADY built, not empty
    <p>Welcome to my app</p>
    <button>Click me</button>
  </body>
</html>
What is "Hydration"?
This is the step people get confused about:


Server sends full HTML
      ↓
Browser shows it instantly (fast!)
      ↓
But buttons/clicks don't work yet
(HTML is static — no JavaScript attached)
      ↓
React JS downloads in background
      ↓
React "hydrates" — looks at existing HTML
and attaches event listeners to it
      ↓
Now useState, onClick, forms all work
Think of it like this:

HTML from server = a photograph of the page (you can see it but can't interact)
Hydration = the photograph comes alive and becomes interactive
The 3 Rendering Types in Next.js
Next.js actually has 3 modes — this is what makes it powerful:

1. SSR — Server Side Rendering

Every request → server builds HTML → sends to browser

// page.jsx — runs on server on EVERY request
export default async function Page() {
  const data = await fetch('https://api.example.com/posts')
  const posts = await data.json()
  return <div>{posts.map(p => <p>{p.title}</p>)}</div>
}
Fresh data every time
Good for: dashboards, feeds, user-specific pages
2. SSG — Static Site Generation

Build time → server builds HTML once → saved as files → sent instantly

// Built ONCE when you run: npm run build
export default async function Page() {
  const data = await fetch('https://api.example.com/posts')
  const posts = await data.json()
  return <div>{posts.map(p => <p>{p.title}</p>)}</div>
}
Super fast — just serving saved HTML files
Good for: blogs, landing pages, docs (content that doesn't change often)
3. CSR — Client Side Rendering (same as React)

"use client"  // ← this tells Next.js: run this in browser

import { useState, useEffect } from "react"

export default function Page() {
  const [data, setData] = useState(null)

  useEffect(() => {
    fetch('/api/posts').then(r => r.json()).then(setData)
  }, [])

  return <div>{data ? data.title : "Loading..."}</div>
}
Runs in browser like normal React
Good for: highly interactive components, user actions
Full Comparison Table
React (Vite)	Next.js SSR	Next.js SSG
Where code runs	Browser	Server	Build time
First page load	Slow (blank screen)	Fast (full HTML)	Fastest (pre-built)
SEO	Bad	Good	Best
Data freshness	Always fresh	Always fresh	Stale until rebuild
Server needed	No	Yes	No
The Big Picture — How Next.js Decides

User requests a page
        ↓
Is it a Server Component? (default)
        ↓ YES
   Runs on server → sends HTML
        ↓
   Has "use client" at top?
        ↓ YES
   Sends HTML shell → hydrates in browser
        ↓
   Page is interactive
The simple rule to remember:

In React (Vite) — browser does all the work
In Next.js — server does the heavy lifting, browser just displays and hydrates
This is why Next.js apps feel faster and rank better on Google — the user never stares at a blank screen waiting for JavaScript to load.
